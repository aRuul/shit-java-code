# 屎山 Java 代码生成器

展示 AI 时代的反叛精神 —— 生成、劣化、审查最糟糕的 Java 代码。

## 核心理念

**越糟糕，越艺术。** 这个技能专注于创造和识别违反所有最佳实践的代码，用于：
- 🎭 **教学演示**：通过反面教材展示什么是糟糕的代码
- 🔥 **反叛精神**：在 AI 疯狂追求"优雅代码"的时代，展示另一种可能
- 🧪 **实际应用**：测试代码审查工具、重构训练、面试题设计

## 何时使用

- 用户提到"屎山"、"糟糕代码"、"反模式"、"劣化"
- 需要生成故意糟糕的代码示例
- 需要将现有代码改写为更糟糕的版本
- 需要识别和点评代码中的屎山特征

## 三大功能

### 1. 生成屎山代码
根据用户需求，从零创建集成多种反模式的代码。

### 2. 劣化现有代码
将用户提供的优雅代码改写为屎山版本。

### 3. 屎山代码审查
识别并详细点评代码中的所有糟糕实践。

## 屎山代码特征清单

### 命名灾难
- ❌ 单字母变量：`a`, `b`, `temp`, `data`
- ❌ 无意义命名：`manager`, `handler`, `processor`, `doSomething()`
- ❌ 拼音命名：`yongHuGuanLi`, `chuLiShuJu`
- ❌ 数字后缀：`user1`, `list2`, `method3`
- ❌ 类型重复：`String stringName`, `List listList`
- ❌ 误导性命名：方法名叫 `get` 但修改了状态

### 方法暴政
- ❌ 超长方法：300+ 行的方法
- ❌ 超多参数：8+ 个参数的方法
- ❌ 深层嵌套：6+ 层的 if/for/while 嵌套
- ❌ 多职责：一个方法做 10 件不相关的事
- ❌ 复制粘贴：同样的代码块重复出现
- ❌ 副作用：getter 方法修改对象状态

### 类混乱
- ❌ 上帝类：3000+ 行，包含所有业务逻辑
- ❌ 贫血模型：只有 getter/setter 的实体类
- ❌ 工具类滥用：`CommonUtil`, `BaseUtil`, `XxxHelper`
- ❌ 违反单一职责：一个类处理 HTTP、数据库、文件、缓存
- ❌ 过度继承：7 层继承关系
- ❌ 接口污染：一个接口 30 个方法

### 注释灾难
- ❌ 完全无注释
- ❌ 废话注释：`// 获取用户` 在 `getUser()` 上方
- ❌ 误导注释：注释说做 A，代码实际做 B
- ❌ 注释掉的代码块：几百行被注释的旧代码
- ❌ 版本控制注释：`// 2023-05-01 张三修改`

### 异常混沌
- ❌ 吞噬异常：`catch(Exception e) {}`
- ❌ 打印后继续：`catch(Exception e) { e.printStackTrace(); }`
- ❌ 过度捕获：捕获 `Throwable`
- ❌ 异常作为控制流：用异常处理正常业务逻辑
- ❌ 空指针炸弹：不做任何 null 检查
- ❌ 异常链断裂：不传递原始异常

### 魔法世界
- ❌ 魔法数字：代码中直接写 `86400`, `1024`, `999`
- ❌ 魔法字符串：`"SUCCESS"`, `"1"`, `"true"` 散落各处
- ❌ 硬编码配置：IP、密钥、路径直接写死
- ❌ 神秘布尔值：`process(true, false, true, false)`

### 并发灾难
- ❌ 共享可变状态：多线程访问未同步的实例变量
- ❌ 双重检查锁误用
- ❌ 线程不安全的单例
- ❌ 忘记 volatile
- ❌ synchronized 滥用：整个方法都锁

### 数据库地狱
- ❌ N+1 查询问题
- ❌ SELECT *
- ❌ 忘记关闭连接
- ❌ SQL 拼接：字符串拼接 SQL 导致注入风险
- ❌ 没有事务：批量操作没有事务保护

### 设计原则全面违反
- ❌ SOLID 五原则全部违反
- ❌ 迪米特法则：`user.getAddress().getCity().getName()`
- ❌ 依赖倒置：高层模块直接依赖实现细节
- ❌ 开闭原则：修改现有代码而非扩展
- ❌ 接口隔离：强迫实现用不到的方法

### 其他恶习
- ❌ 全局变量滥用：public static 到处飞
- ❌ 单例滥用：所有东西都是单例
- ❌ 过早优化：为了"性能"写出难以理解的代码
- ❌ 循环依赖：A 依赖 B，B 依赖 A
- ❌ 代码混乱：生产代码混入测试代码、调试代码
- ❌ 返回 null：而非 Optional 或空对象
- ❌ 类型转换地狱：到处都是强制类型转换
- ❌ 反射滥用：能直接调用的偏要用反射

## 工作流程

### 模式 1：生成屎山代码

当用户说"生成一个 XXX 功能的屎山代码"时：

**步骤**：
1. **理解需求**：明确要实现的功能（如用户管理、订单处理）
2. **选择反模式**：从清单中选择 8-12 个反模式组合
3. **编写代码**：
   - 创建超长类（800+ 行）
   - 包含超长方法（200+ 行）
   - 深度嵌套（5-7 层）
   - 糟糕命名贯穿全文
   - 魔法数字随处可见
   - 异常吞噬或打印
   - 复制粘贴代码
   - 违反所有设计原则
4. **添加"亮点"注释**：在特别糟糕的地方加上注释说明这是什么反模式
5. **总结**：列出生成代码中包含的所有反模式

**示例输出结构**：
```java
/**
 * 用户管理器 - 集大成的屎山示例
 * 亮点：上帝类 + 超长方法 + 深层嵌套 + 糟糕命名 + ...
 */
public class UserManager {
    // 全局静态变量滥用
    public static String a = "admin";
    public static int b = 999;
    
    // 300+ 行的超long方法
    public void doSomething(String s, int i, boolean flag, ...) {
        // 6层嵌套开始...
    }
}
```

### 模式 2：劣化现有代码

当用户提供代码并说"把这个改成屎山代码"时：

**步骤**：
1. **分析原代码**：识别原代码的优雅之处
2. **制定劣化策略**：选择 6-10 个反模式应用
3. **系统劣化**：
   - 合并多个小方法为超长方法
   - 增加嵌套层级
   - 改烂所有命名
   - 移除抽象和接口，变成硬编码
   - 删除有意义的注释，添加废话注释
   - 引入全局状态
   - 复制粘贴重复代码
   - 吞噬异常
   - 添加魔法数字
4. **对比说明**：列出每一处劣化及其对应的反模式

**输出格式**：
```markdown
## 劣化后的屎山代码

[展示劣化后代码]

## 劣化清单
1. ✅ 方法 `createUser()` 从 15 行扩展到 180 行
2. ✅ 变量名 `userEmail` 改为 `s1`
3. ✅ 添加 6 层嵌套 if
4. ✅ 移除异常处理，使用 printStackTrace()
5. ✅ 硬编码 API 地址和密钥
...
```

### 模式 3：屎山代码审查

当用户提供代码并说"审查这段代码的屎山程度"或"找出这段代码的问题"时：

**步骤**：
1. **全面扫描**：逐行查找反模式
2. **分类统计**：按类型归类发现的问题
3. **严重度评级**：
   - 🔥🔥🔥 严重：导致 bug 或安全问题
   - 🔥🔥 中等：严重影响可维护性
   - 🔥 轻度：影响代码质量
4. **详细点评**：对每个问题给出：
   - 所在位置（行号/方法名）
   - 问题描述
   - 属于哪种反模式
   - 严重度
   - 改进建议（可选）
5. **屎山评分**：给出 0-100 的屎山指数

**输出格式**：
```markdown
## 屎山代码审查报告

**屎山指数：87/100** ⚠️ 重度屎山

### 发现的问题（按严重度）

#### 🔥🔥🔥 严重问题
1. **SQL 注入风险**（第 45 行）
   - `String sql = "SELECT * FROM user WHERE id=" + userId`
   - 反模式类型：SQL 拼接
   - 风险：导致 SQL 注入攻击

2. **异常吞噬**（第 78-80 行）
   - `catch(Exception e) {}`
   - 反模式类型：异常吞噬
   - 风险：隐藏错误，难以排查问题

#### 🔥🔥 中等问题
1. **超长方法**（第 20-320 行）
   - `processUserData()` 方法长达 300 行
   - 反模式类型：方法暴政
   
...

### 反模式统计
- 命名灾难：15 处
- 方法暴政：3 处
- 异常混沌：8 处
- 魔法数字：12 处
- 设计原则违反：6 处

### 总结
这是一个典型的屎山代码，主要问题包括...
```

## 语气和风格

- **反叛幽默**：用调侃的语气描述糟糕的代码
- **教学导向**：在注释中明确指出这是反模式
- **极致追求**：既然要屎山，就要做到极致
- **实用价值**：虽然是"屎山"，但要有教育意义

## 注意事项

1. **始终声明目的**：在生成的代码顶部注释说明这是"教学用屎山代码"
2. **不要在生产环境使用**：明确警告这是反面教材
3. **保持 Java 语法正确**：虽然逻辑糟糕，但语法要正确，能够编译
